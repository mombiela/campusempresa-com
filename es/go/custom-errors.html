Introducción a los errores personalizados

En Go, el manejo de errores es una parte crucial para escribir código robusto y mantenible. Aunque el tipo error incorporado en Go suele ser suficiente, hay escenarios donde los errores personalizados pueden proporcionar más contexto y mejorar la depuración. Esta sección cubrirá cómo crear y usar errores personalizados en Go.

Creación de errores personalizados

Para crear un error personalizado en Go, normalmente defines un nuevo tipo que implementa la interfaz error. La interfaz error en Go es simple y consta de un solo método:

type error interface {
    Error() string
}

Definición de un tipo de error personalizado

Aquí tienes cómo puedes definir un tipo de error personalizado:

package main

import (
    "fmt"
)

// Define un tipo de error personalizado
type MyCustomError struct {
    Code    int
    Message string
}

// Implementa el método Error() para el tipo de error personalizado
func (e *MyCustomError) Error() string {
    return fmt.Sprintf("Error %d: %s", e.Code, e.Message)
}

func main() {
    // Crea una instancia del error personalizado
    err := &MyCustomError{
        Code:    404,
        Message: "Resource not found",
    }

    // Imprime el error
    fmt.Println(err)
}

Explicación

  • Definición de la estructura: MyCustomError es una estructura con campos Code y Message.
  • Método Error: El método Error() formatea el mensaje de error, haciendo que MyCustomError satisfaga la interfaz error.
  • Uso: Se crea e imprime una instancia de MyCustomError, demostrando cómo se puede usar el error personalizado.

Uso de errores personalizados

Los errores personalizados se pueden usar como cualquier otro error en Go. Pueden ser devueltos desde funciones y comprobados usando afirmaciones de tipo o conmutadores de tipo.

Devolución de errores personalizados desde funciones

Aquí tienes un ejemplo de una función que devuelve un error personalizado:

package main

import (
    "fmt"
)

// Define un tipo de error personalizado
type MyCustomError struct {
    Code    int
    Message string
}

func (e *MyCustomError) Error() string {
    return fmt.Sprintf("Error %d: %s", e.Code, e.Message)
}

// Función que devuelve un error personalizado
func doSomething() error {
    return &MyCustomError{
        Code:    500,
        Message: "Internal Server Error",
    }
}

func main() {
    err := doSomething()
    if err != nil {
        fmt.Println(err)
    }
}

Comprobación de errores personalizados específicos

Puedes usar afirmaciones de tipo o conmutadores de tipo para comprobar errores personalizados específicos:

package main

import (
    "fmt"
)

// Define un tipo de error personalizado
type MyCustomError struct {
    Code    int
    Message string
}

func (e *MyCustomError) Error() string {
    return fmt.Sprintf("Error %d: %s", e.Code, e.Message)
}

func doSomething() error {
    return &MyCustomError{
        Code:    500,
        Message: "Internal Server Error",
    }
}

func main() {
    err := doSomething()
    if err != nil {
        // Afirmación de tipo
        if customErr, ok := err.(*MyCustomError); ok {
            fmt.Printf("Ocurrió un error personalizado: Code=%d, Message=%s\n", customErr.Code, customErr.Message)
        } else {
            fmt.Println("Ocurrió un error:", err)
        }
    }
}

Mejores prácticas para errores personalizados

  • Usa errores personalizados con moderación: Solo usa errores personalizados cuando sea necesario un contexto adicional.
  • Mensajes de error consistentes: Asegúrate de que los mensajes de error sean consistentes e informativos.
  • Envolvimiento de errores: Considera usar las características de envolvimiento de errores de Go 1.13+ (fmt.Errorf con %w) para escenarios de manejo de errores más complejos.

Conclusión

Los errores personalizados en Go proporcionan una forma poderosa de agregar contexto al manejo de errores. Al definir tipos de errores personalizados e implementar la interfaz error, puedes crear mensajes de error más informativos y mantenibles. Recuerda usar los errores personalizados con juicio y seguir las mejores prácticas para asegurar que tu código permanezca limpio y comprensible.
