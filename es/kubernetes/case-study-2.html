Introducción

En este estudio de caso, exploraremos cómo escalar aplicaciones utilizando Kubernetes. El escalado es un aspecto crítico de la gestión de aplicaciones en producción, asegurando que tu aplicación pueda manejar cargas variables de manera eficiente. Cubriremos tanto el escalado horizontal como el vertical, y proporcionaremos ejemplos prácticos para ayudarte a entender los conceptos.

Conceptos Clave

  • Escalado Horizontal: Añadir más instancias de tu aplicación para manejar una carga aumentada.
  • Escalado Vertical: Aumentar los recursos (CPU, memoria) de las instancias existentes para manejar una carga aumentada.
  • Autoscaler Horizontal de Pods de Kubernetes (HPA): Escala automáticamente el número de pods en un despliegue basado en la utilización observada de la CPU u otras métricas seleccionadas.
  • Solicitudes y Límites de Recursos: Definen los recursos mínimos y máximos que un contenedor puede usar.

Escalado Horizontal

Entendiendo el Escalado Horizontal

El escalado horizontal implica añadir más instancias (pods) de tu aplicación para distribuir la carga. Esto a menudo es más rentable y proporciona mejor tolerancia a fallos en comparación con el escalado vertical.

Ejemplo: Configuración del Autoscaler Horizontal de Pods (HPA)

Vamos a crear un despliegue y configurar HPA para escalarlo basado en el uso de la CPU.

  1. Crear un Despliegue:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app-image
        resources:
          requests:
            cpu: "100m"
          limits:
            cpu: "500m"

  2. Aplicar el Despliegue:

    kubectl apply -f deployment.yaml

  3. Crear un HPA:

    apiVersion: autoscaling/v1
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 1
  maxReplicas: 10
  targetCPUUtilizationPercentage: 50

  4. Aplicar el HPA:

    kubectl apply -f hpa.yaml

Explicación

  • Despliegue: Define una sola réplica de my-app con solicitudes y límites de CPU.
  • HPA: Monitorea la utilización de la CPU del despliegue my-app y escala el número de pods entre 1 y 10 para mantener una utilización promedio de CPU del 50%.

Escalado Vertical

Entendiendo el Escalado Vertical

El escalado vertical implica aumentar los límites de recursos de los pods existentes. Esto puede ser útil cuando tu aplicación necesita más recursos para manejar una carga aumentada.

Ejemplo: Ajuste de Límites de Recursos

  1. Actualizar el Despliegue:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app-image
        resources:
          requests:
            cpu: "200m"
            memory: "256Mi"
          limits:
            cpu: "1"
            memory: "512Mi"

  2. Aplicar el Despliegue Actualizado:

    kubectl apply -f deployment.yaml

Explicación

  • Solicitudes de Recursos: Los recursos mínimos garantizados para el contenedor.
  • Límites de Recursos: Los recursos máximos que el contenedor puede usar.

Comparación del Escalado Horizontal y Vertical

| Aspecto | Escalado Horizontal | Escalado Vertical | |-----------------------|------------------------------------------|-------------------------------------------| | Enfoque | Añadir más instancias (pods) | Aumentar recursos de instancias existentes| | Costo | Generalmente más rentable | Puede ser costoso | | Tolerancia a Fallos | Mayor | Menor | | Complejidad | Requiere balanceo de carga | Más simple | | Caso de Uso | Manejar tráfico aumentado | Manejar tareas intensivas en recursos |

Conclusión

Escalar aplicaciones en Kubernetes es esencial para mantener el rendimiento y la fiabilidad bajo cargas variables. El escalado horizontal, logrado a través de herramientas como el Autoscaler Horizontal de Pods, es generalmente más rentable y proporciona mejor tolerancia a fallos. El escalado vertical, por otro lado, es útil para aplicaciones intensivas en recursos pero puede ser más costoso. Entender cuándo y cómo usar estas estrategias de escalado te ayudará a gestionar tus aplicaciones de manera más efectiva en un entorno de Kubernetes.
