Introducción

Las actualizaciones continuas y las reversiones son conceptos esenciales en Kubernetes para gestionar despliegues de aplicaciones. Permiten actualizar tus aplicaciones sin tiempo de inactividad y volver a versiones anteriores si algo sale mal.

Conceptos Clave

  • Actualización Continua: Una estrategia de despliegue que actualiza pods de manera incremental para asegurar un tiempo de inactividad mínimo.
  • Reversión: El proceso de volver a una versión de despliegue anterior si la actual falla o tiene problemas.
  • Despliegue: Un recurso de Kubernetes que gestiona un conjunto de pods idénticos, asegurando que un número especificado de ellos esté en ejecución.

Actualizaciones Continuas

Cómo Funcionan las Actualizaciones Continuas

Durante una actualización continua, Kubernetes reemplaza de manera incremental los pods antiguos con nuevos. Esto asegura que tu aplicación permanezca disponible durante todo el proceso de actualización.

Pasos para Realizar una Actualización Continua

  1. Actualizar el Despliegue: Modificar la configuración del despliegue para especificar la nueva versión de la aplicación.
  2. Aplicar la Actualización: Usar kubectl apply para aplicar los cambios.
  3. Monitorear la Actualización: Verificar el estado del despliegue para asegurar que la actualización esté procediendo como se espera.

Ejemplo: Realizando una Actualización Continua

Supongamos que tienes un despliegue llamado my-app y quieres actualizar su imagen a una nueva versión.

  1. Ver la configuración actual del despliegue:

    kubectl get deployment my-app -o yaml

  2. Actualizar el despliegue para usar la nueva imagen:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app-container
        image: my-app:2.0  # Actualiza la versión de la imagen aquí

  3. Aplicar la configuración actualizada:

    kubectl apply -f my-app-deployment.yaml

  4. Monitorear el estado del despliegue:

    kubectl rollout status deployment/my-app

Estrategias de Actualización Continua

  • Max Surge: El número máximo de pods que pueden ser creados por encima del número deseado de pods durante la actualización.
  • Max Unavailable: El número máximo de pods que pueden estar no disponibles durante la actualización.

Ejemplo de configuración:

spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1

Reversiones

Cómo Funcionan las Reversiones

Una reversión revierte el despliegue a una versión anterior si la versión actual tiene problemas. Kubernetes mantiene un historial de despliegues anteriores, lo que facilita volver a un estado estable.

Pasos para Realizar una Reversión

  1. Verificar el Historial de Despliegue: Ver el historial de despliegues anteriores.
  2. Revertir a una Revisión Anterior: Usar kubectl rollout undo para volver a una revisión específica.
  3. Monitorear la Reversión: Asegurarse de que la reversión sea exitosa y que la aplicación esté estable.

Ejemplo: Realizando una Reversión

  1. Ver el historial del despliegue:

    kubectl rollout history deployment/my-app

  2. Revertir a una revisión anterior:

    kubectl rollout undo deployment/my-app --to-revision=2

  3. Monitorear el estado de la reversión:

    kubectl rollout status deployment/my-app

Conclusión

Las actualizaciones continuas y las reversiones son características poderosas en Kubernetes que ayudan a asegurar que tus aplicaciones se actualicen sin problemas y puedan revertirse rápidamente si surgen problemas. Al entender y utilizar estas características, puedes mantener una alta disponibilidad y fiabilidad para tus aplicaciones.
