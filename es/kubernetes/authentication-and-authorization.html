Introducción

La autenticación y la autorización son componentes críticos para asegurar un clúster de Kubernetes. La autenticación verifica la identidad de un usuario o servicio, mientras que la autorización determina qué acciones se les permite realizar. Esta sección te guiará desde los conceptos básicos hasta los avanzados de autenticación y autorización en Kubernetes.

Autenticación

Descripción General

La autenticación en Kubernetes es el proceso de verificar la identidad de un usuario o servicio. Kubernetes admite múltiples métodos de autenticación, incluyendo:

  • Certificados de Cliente
  • Tokens Bearer
  • Tokens OpenID Connect
  • Autenticación de Token Webhook

Certificados de Cliente

Los certificados de cliente se utilizan para autenticar usuarios y servicios. Kubernetes utiliza certificados TLS para asegurar la comunicación entre el cliente y el servidor API.

Ejemplo

apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority: /path/to/ca.crt
    server: https://kubernetes.example.com
  name: example-cluster
contexts:
- context:
    cluster: example-cluster
    user: example-user
  name: example-context
current-context: example-context
users:
- name: example-user
  user:
    client-certificate: /path/to/client.crt
    client-key: /path/to/client.key

Tokens Bearer

Los tokens Bearer son un tipo de token que se puede usar para autenticar solicitudes al servidor API de Kubernetes. Estos tokens se utilizan típicamente para cuentas de servicio.

Ejemplo

apiVersion: v1
kind: ServiceAccount
metadata:
  name: example-service-account

Tokens OpenID Connect

Los tokens OpenID Connect (OIDC) se utilizan para autenticar usuarios a través de un proveedor de identidad externo.

Ejemplo

apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority: /path/to/ca.crt
    server: https://kubernetes.example.com
  name: example-cluster
contexts:
- context:
    cluster: example-cluster
    user: oidc-user
  name: example-context
current-context: example-context
users:
- name: oidc-user
  user:
    auth-provider:
      name: oidc
      config:
        client-id: your-client-id
        client-secret: your-client-secret
        id-token: your-id-token
        idp-issuer-url: https://accounts.google.com
        refresh-token: your-refresh-token

Autenticación de Token Webhook

La autenticación de token webhook te permite integrarte con un servicio de autenticación externo.

Ejemplo

apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority: /path/to/ca.crt
    server: https://kubernetes.example.com
  name: example-cluster
contexts:
- context:
    cluster: example-cluster
    user: webhook-user
  name: example-context
current-context: example-context
users:
- name: webhook-user
  user:
    token: your-webhook-token

Autorización

Descripción General

La autorización en Kubernetes determina qué acciones se le permite realizar a un usuario o servicio. Kubernetes admite varios métodos de autorización, incluyendo:

  • Control de Acceso Basado en Roles (RBAC)
  • Control de Acceso Basado en Atributos (ABAC)
  • Autorización Webhook

Control de Acceso Basado en Roles (RBAC)

RBAC es el método de autorización más comúnmente utilizado en Kubernetes. Utiliza roles y enlaces de roles para controlar el acceso a los recursos.

Ejemplo

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: example-role
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: example-rolebinding
  namespace: default
subjects:
- kind: User
  name: example-user
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: example-role
  apiGroup: rbac.authorization.k8s.io

Control de Acceso Basado en Atributos (ABAC)

ABAC utiliza políticas definidas en JSON para controlar el acceso a los recursos.

Ejemplo

{
  "apiVersion": "abac.authorization.kubernetes.io/v1beta1",
  "kind": "Policy",
  "spec": {
    "user": "example-user",
    "namespace": "default",
    "resource": "pods",
    "readonly": true
  }
}

Autorización Webhook

La autorización webhook te permite integrarte con un servicio de autorización externo.

Ejemplo

apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority: /path/to/ca.crt
    server: https://kubernetes.example.com
  name: example-cluster
contexts:
- context:
    cluster: example-cluster
    user: webhook-user
  name: example-context
current-context: example-context
users:
- name: webhook-user
  user:
    token: your-webhook-token

Conclusión

La autenticación y la autorización son esenciales para asegurar un clúster de Kubernetes. Al comprender e implementar los diversos métodos disponibles, puedes asegurarte de que solo los usuarios y servicios autorizados tengan acceso a tus recursos. Esta sección cubrió desde los conceptos básicos hasta los avanzados, incluyendo certificados de cliente, tokens Bearer, tokens OpenID Connect, autenticación de token webhook, RBAC, ABAC y autorización webhook.
