Objetivos de la Presentación Final

La presentación final tiene como objetivo que los estudiantes demuestren su comprensión y aplicación de los conceptos aprendidos a lo largo del curso de Arquitecturas de Sistemas. Los estudiantes deberán:

  • Presentar un proyecto de arquitectura de sistema completo.
  • Explicar las decisiones de diseño tomadas.
  • Mostrar cómo se implementan los principios de arquitectura en su proyecto.
  • Responder preguntas y recibir retroalimentación.

Estructura de la Presentación

Cada presentación debe seguir una estructura clara y lógica para asegurar que todos los aspectos importantes sean cubiertos. La estructura recomendada es la siguiente:

  1. Introducción

    • Breve descripción del proyecto.
    • Objetivos del sistema.
    • Público objetivo y necesidades del usuario.

  2. Arquitectura del Sistema

    • Diagrama de arquitectura general.
    • Componentes principales y sus interacciones.
    • Justificación de la elección de la arquitectura (e.g., microservicios, monolítica, etc.).

  3. Tecnologías Utilizadas

    • Lenguajes de programación.
    • Frameworks y bibliotecas.
    • Herramientas de desarrollo y despliegue.

  4. Patrones de Diseño

    • Patrones de diseño aplicados (e.g., Singleton, Factory, Observer).
    • Ejemplos de código que demuestren la implementación de estos patrones.

  5. Seguridad y Escalabilidad

    • Estrategias de seguridad implementadas.
    • Planes de escalabilidad y cómo se manejan las cargas de trabajo.

  6. Pruebas y Validación

    • Métodos de prueba utilizados (e.g., unit tests, integration tests).
    • Resultados de las pruebas y cómo se validó el sistema.

  7. Conclusión

    • Resumen de los puntos clave.
    • Lecciones aprendidas.
    • Posibles mejoras futuras.

Ejemplo de Proyecto

Introducción

El proyecto presentado es un sistema de gestión de inventarios para una tienda en línea. El objetivo del sistema es permitir a los administradores gestionar el inventario de productos, mientras que los clientes pueden ver la disponibilidad de los productos en tiempo real.

Arquitectura del Sistema

  • Diagrama de Arquitectura General:

  • Componentes Principales:

    • Frontend: Aplicación web desarrollada en React.
    • Backend: API RESTful construida con Node.js y Express.
    • Base de Datos: MongoDB para almacenamiento de datos.

  • Interacciones:

    • El frontend se comunica con el backend a través de llamadas API.
    • El backend interactúa con la base de datos para CRUD (Create, Read, Update, Delete) operaciones.

Tecnologías Utilizadas

  • Lenguajes de Programación:

    • JavaScript (Node.js, React)

  • Frameworks y Bibliotecas:

    • Express para el backend.
    • Mongoose para la interacción con MongoDB.
    • Redux para el manejo del estado en el frontend.

  • Herramientas de Desarrollo y Despliegue:

    • Docker para la contenedorización.
    • Jenkins para CI/CD.
    • AWS para el despliegue en la nube.

Patrones de Diseño

  • Singleton:

    class Database {
  constructor() {
    if (!Database.instance) {
      this.connection = this.createConnection();
      Database.instance = this;
    }
    return Database.instance;
  }

  createConnection() {
    // Lógica para crear una conexión a la base de datos
  }
}

const instance = new Database();
Object.freeze(instance);
export default instance;

  • Factory:

    class ProductFactory {
  static createProduct(type, attributes) {
    switch (type) {
      case 'electronic':
        return new ElectronicProduct(attributes);
      case 'furniture':
        return new FurnitureProduct(attributes);
      default:
        throw new Error('Tipo de producto no soportado');
    }
  }
}

Seguridad y Escalabilidad

  • Estrategias de Seguridad:

    • Autenticación y autorización con JWT.
    • Encriptación de datos sensibles.
    • Validación de entradas para prevenir inyecciones SQL y XSS.

  • Planes de Escalabilidad:

    • Uso de balanceadores de carga.
    • Implementación de microservicios para componentes críticos.
    • Escalado horizontal de la base de datos.

Pruebas y Validación

  • Métodos de Prueba:

    • Pruebas unitarias con Jest.
    • Pruebas de integración con Supertest.
    • Pruebas de carga con JMeter.

  • Resultados de las Pruebas:

    • 95% de cobertura de código.
    • Todas las pruebas de integración pasaron.
    • El sistema soporta hasta 1000 solicitudes por segundo sin degradación del rendimiento.

Conclusión

En la presentación final, los estudiantes deben demostrar su capacidad para diseñar y construir una arquitectura de sistema robusta y escalable. Deben ser capaces de justificar sus decisiones de diseño y mostrar cómo han aplicado los principios y patrones de arquitectura aprendidos durante el curso. La retroalimentación recibida durante la presentación será crucial para su desarrollo profesional y les ayudará a identificar áreas de mejora para futuros proyectos.
