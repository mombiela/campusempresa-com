Introducción a las Teorías en JUnit

Las teorías en JUnit son una característica poderosa que te permite probar tu código con una amplia gama de valores de entrada. A diferencia de las pruebas unitarias tradicionales que prueban un método con un conjunto específico de entradas, las teorías prueban el método con múltiples conjuntos de entradas para asegurar que se comporte correctamente bajo diversas condiciones.

Conceptos Clave

  • Teoría: Un tipo especial de prueba que está parametrizada y puede ejecutarse con múltiples conjuntos de datos.
  • Puntos de Datos: Los conjuntos de datos que se utilizan para ejecutar la teoría.
  • Suposiciones: Condiciones que deben ser verdaderas para que la teoría se ejecute.

Configuración de Teorías

Para usar teorías en JUnit, necesitas incluir la biblioteca junit-theories en tu proyecto. Si estás usando Maven, agrega la siguiente dependencia a tu pom.xml:

<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13.2</version>
</dependency>

Escribiendo una Teoría Simple

Aquí tienes un ejemplo simple de una teoría en JUnit:

import org.junit.experimental.theories.Theories;
import org.junit.experimental.theories.Theory;
import org.junit.experimental.theories.DataPoint;
import org.junit.runner.RunWith;
import static org.junit.Assume.assumeTrue;
import static org.junit.Assert.assertTrue;

@RunWith(Theories.class)
public class SimpleTheoryTest {

    @DataPoint
    public static int positiveNumber = 1;

    @DataPoint
    public static int negativeNumber = -1;

    @Theory
    public void testAbsoluteValue(int number) {
        assumeTrue(number != 0); // Omitir la prueba si el número es cero
        assertTrue(Math.abs(number) >= 0);
    }
}

Explicación

  • @RunWith(Theories.class): Esta anotación le dice a JUnit que ejecute la clase como una prueba de teoría.
  • @DataPoint: Esta anotación marca los puntos de datos que se utilizarán en la teoría.
  • @Theory: Esta anotación marca el método como una teoría.
  • assumeTrue: Este método se utiliza para omitir la prueba si la suposición es falsa.

Usando Múltiples Puntos de Datos

Puedes definir múltiples puntos de datos para probar tu teoría con una gama más amplia de entradas:

@RunWith(Theories.class)
public class MultipleDataPointsTest {

    @DataPoint
    public static int positiveNumber = 1;

    @DataPoint
    public static int negativeNumber = -1;

    @DataPoint
    public static int zero = 0;

    @Theory
    public void testNonNegativeAbsoluteValue(int number) {
        assertTrue(Math.abs(number) >= 0);
    }
}

Combinando Puntos de Datos

También puedes combinar múltiples puntos de datos para probar escenarios más complejos:

@RunWith(Theories.class)
public class CombinedDataPointsTest {

    @DataPoint
    public static int positiveNumber = 1;

    @DataPoint
    public static int negativeNumber = -1;

    @DataPoint
    public static int zero = 0;

    @DataPoint
    public static String nonEmptyString = "JUnit";

    @DataPoint
    public static String emptyString = "";

    @Theory
    public void testStringLength(String str, int number) {
        assumeTrue(str != null);
        assertTrue(str.length() >= 0);
    }
}

Suposiciones en Teorías

Las suposiciones se utilizan para filtrar puntos de datos que no cumplen ciertos criterios. Si una suposición falla, la teoría se omite para ese conjunto de puntos de datos:

@RunWith(Theories.class)
public class AssumptionsTest {

    @DataPoint
    public static int positiveNumber = 1;

    @DataPoint
    public static int negativeNumber = -1;

    @DataPoint
    public static int zero = 0;

    @Theory
    public void testPositiveNumbers(int number) {
        assumeTrue(number > 0); // Solo probar números positivos
        assertTrue(number > 0);
    }
}

Conclusión

Las teorías en JUnit proporcionan una forma flexible de probar tu código con una variedad de entradas, asegurando que se comporte correctamente bajo diferentes condiciones. Al usar puntos de datos y suposiciones, puedes crear pruebas exhaustivas que cubran una amplia gama de escenarios. Este enfoque ayuda a identificar casos límite y posibles errores que podrían no ser detectados con pruebas unitarias tradicionales.
