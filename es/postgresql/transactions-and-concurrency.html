Introducción a las Transacciones

Las transacciones son un concepto fundamental en las bases de datos, asegurando que una serie de operaciones se completen exitosamente juntas o fallen juntas, manteniendo la integridad de los datos.

  • Atomicidad: Asegura que todas las operaciones dentro de una transacción se completen; si no, la transacción se aborta.
  • Consistencia: Asegura que una transacción lleve la base de datos de un estado válido a otro.
  • Aislamiento: Asegura que las transacciones se procesen de manera segura e independiente al mismo tiempo sin interferencia.
  • Durabilidad: Asegura que una vez que una transacción ha sido confirmada, permanecerá así, incluso en caso de una falla del sistema.

Comandos Básicos de Transacción

  • BEGIN: Inicia una nueva transacción.
  • COMMIT: Guarda todos los cambios realizados durante la transacción.
  • ROLLBACK: Revierte todos los cambios realizados durante la transacción.

Ejemplo

BEGIN;

INSERT INTO accounts (user_id, balance) VALUES (1, 1000);
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;

COMMIT;

En este ejemplo:

  1. Se inicia una transacción con BEGIN.
  2. Se inserta una nueva cuenta.
  3. Se actualiza el saldo de la cuenta.
  4. La transacción se confirma, haciendo permanentes todos los cambios.

Control de Concurrencia

El control de concurrencia es esencial para asegurar que las transacciones de la base de datos se realicen concurrentemente sin violar la integridad de los datos.

Niveles de Aislamiento

PostgreSQL soporta varios niveles de aislamiento que controlan la visibilidad de los cambios realizados por transacciones concurrentes.

  • Read Uncommitted: Las transacciones pueden ver cambios no confirmados realizados por otras transacciones.
  • Read Committed: Una transacción solo ve los cambios confirmados realizados por otras transacciones.
  • Repeatable Read: Asegura que si una transacción lee un valor, verá el mismo valor si lo vuelve a leer.
  • Serializable: Asegura un aislamiento completo de otras transacciones, proporcionando el nivel más alto de aislamiento.

Ejemplo

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

BEGIN;

SELECT balance FROM accounts WHERE user_id = 1;

-- Realizar algunas operaciones

COMMIT;

En este ejemplo:

  1. El nivel de aislamiento de la transacción se establece en REPEATABLE READ.
  2. Se inicia una transacción.
  3. Se lee el saldo de la tabla accounts.
  4. La transacción se confirma.

Mecanismos de Bloqueo

PostgreSQL utiliza bloqueos para gestionar el acceso concurrente a los datos.

  • Bloqueos a nivel de fila: Bloquean filas individuales.
  • Bloqueos a nivel de tabla: Bloquean tablas enteras.

Ejemplo

BEGIN;

SELECT * FROM accounts WHERE user_id = 1 FOR UPDATE;

-- Realizar algunas operaciones

COMMIT;

En este ejemplo:

  1. Se inicia una transacción.
  2. Se adquiere un bloqueo a nivel de fila en la fila seleccionada usando FOR UPDATE.
  3. Se realizan operaciones.
  4. La transacción se confirma.

Interbloqueos

Los interbloqueos ocurren cuando dos o más transacciones están esperando que la otra libere bloqueos, causando un ciclo de dependencias.

Ejemplo

-- Transacción 1
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;

-- Transacción 2
BEGIN;
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;

-- Transacción 1
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2; -- espera a Transacción 2

-- Transacción 2
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1; -- espera a Transacción 1

En este ejemplo:

  1. La Transacción 1 bloquea una fila.
  2. La Transacción 2 bloquea otra fila.
  3. Ambas transacciones luego esperan que la otra libere sus bloqueos, causando un interbloqueo.

Conclusión

Entender las transacciones y el control de concurrencia es crucial para mantener la integridad de los datos y el rendimiento en PostgreSQL. Al dominar los comandos de transacción, los niveles de aislamiento, los mecanismos de bloqueo y el manejo de interbloqueos, puedes asegurar que tus operaciones de base de datos sean confiables y eficientes.
