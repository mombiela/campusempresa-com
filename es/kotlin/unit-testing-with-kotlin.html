Introducción a las Pruebas Unitarias

Las pruebas unitarias son un método de prueba de software donde se prueban unidades o componentes individuales de un software. El propósito es validar que cada unidad del software funcione como se espera. En Kotlin, las pruebas unitarias se realizan típicamente usando el marco JUnit.

Conceptos Clave

  • Prueba Unitaria: Una prueba que verifica una única "unidad" de código, generalmente una función o método.
  • Caso de Prueba: Un conjunto de condiciones o variables bajo las cuales un probador determina si una unidad de código funciona como se espera.
  • Suite de Pruebas: Una colección de casos de prueba.
  • Mocking: Crear una versión falsa de un objeto para probar interacciones entre objetos.

Configuración de JUnit en Kotlin

Para comenzar con las pruebas unitarias en Kotlin, necesitas configurar JUnit en tu proyecto. Esto se puede hacer agregando las dependencias necesarias a tu archivo build.gradle.

dependencies {
    testImplementation 'org.junit.jupiter:junit-jupiter-api:5.7.0'
    testRuntimeOnly 'org.junit.jupiter:junit-jupiter-engine:5.7.0'
}

Escribiendo Tu Primera Prueba Unitaria

Vamos a escribir una simple prueba unitaria para una función en Kotlin. Supongamos que tenemos una función que suma dos números:

fun add(a: Int, b: Int): Int {
    return a + b
}

Para probar esta función, podemos escribir una prueba unitaria usando JUnit:

import org.junit.jupiter.api.Assertions.assertEquals
import org.junit.jupiter.api.Test

class MathTests {

    @Test
    fun testAdd() {
        val result = add(2, 3)
        assertEquals(5, result)
    }
}

Explicación

  • @Test: Esta anotación le dice a JUnit que la función siguiente es un caso de prueba.
  • assertEquals: Esta función verifica si el valor esperado (5) es igual al valor actual (result).

Técnicas Avanzadas de Pruebas Unitarias

A medida que te sientas más cómodo con las pruebas unitarias básicas, puedes explorar técnicas más avanzadas como el mocking y las pruebas parametrizadas.

Mocking con Mockito

Mockito es una biblioteca popular para crear objetos mock en pruebas unitarias. Para usar Mockito, agrega la siguiente dependencia a tu archivo build.gradle:

testImplementation 'org.mockito:mockito-core:3.7.7'

Aquí tienes un ejemplo de cómo usar Mockito para mockear una dependencia:

import org.junit.jupiter.api.Test
import org.mockito.Mockito.*

class UserServiceTest {

    @Test
    fun testGetUser() {
        val mockRepository = mock(UserRepository::class.java)
        `when`(mockRepository.getUser(1)).thenReturn(User(1, "John Doe"))

        val userService = UserService(mockRepository)
        val user = userService.getUser(1)

        assertEquals("John Doe", user.name)
    }
}

Pruebas Parametrizadas

Las pruebas parametrizadas te permiten ejecutar la misma prueba con diferentes entradas. JUnit 5 proporciona soporte para pruebas parametrizadas.

import org.junit.jupiter.params.ParameterizedTest
import org.junit.jupiter.params.provider.CsvSource
import org.junit.jupiter.api.Assertions.assertEquals

class MathTests {

    @ParameterizedTest
    @CsvSource("1, 1, 2", "2, 3, 5", "3, 5, 8")
    fun testAdd(a: Int, b: Int, expected: Int) {
        assertEquals(expected, add(a, b))
    }
}

Mejores Prácticas para Pruebas Unitarias

  • Escribe Pruebas Independientes: Cada prueba debe ser independiente de las demás.
  • Usa Nombres de Pruebas Significativos: Los nombres de las pruebas deben describir claramente lo que están probando.
  • Prueba Casos Límite: Asegúrate de probar casos límite y no solo el "camino feliz".
  • Mantén las Pruebas Rápidas: Las pruebas unitarias deben ser rápidas de ejecutar.

Conclusión

Las pruebas unitarias son una parte crucial del desarrollo de software que asegura que tu código funcione como se espera. Usando JUnit con Kotlin, puedes escribir pruebas unitarias efectivas y eficientes. Comienza con pruebas simples y gradualmente incorpora técnicas avanzadas como el mocking y las pruebas parametrizadas. Seguir las mejores prácticas te ayudará a mantener una base de código robusta y confiable.
