Introducción a las Pruebas en Kotlin

Las pruebas son una parte crucial del desarrollo de software que asegura que tu código funcione como se espera. En Kotlin, puedes usar varios marcos de pruebas como JUnit, TestNG y KotlinTest. Esta sección te guiará a través de las mejores prácticas para escribir pruebas efectivas en Kotlin.

Tipos de Pruebas

  • Pruebas Unitarias: Prueban componentes o funciones individuales.
  • Pruebas de Integración: Prueban la interacción entre diferentes componentes.
  • Pruebas de Extremo a Extremo: Prueban el flujo completo de la aplicación.

Configuración de tu Entorno de Pruebas

Para comenzar con las pruebas en Kotlin, necesitas configurar tu entorno. Aquí tienes una configuración básica usando JUnit:

  1. Agrega dependencias a tu archivo build.gradle.kts:

    dependencies {
    testImplementation("org.junit.jupiter:junit-jupiter-api:5.7.0")
    testRuntimeOnly("org.junit.jupiter:junit-jupiter-engine:5.7.0")
}

  2. Crea una clase de prueba:

    import org.junit.jupiter.api.Assertions.*
import org.junit.jupiter.api.Test

class MyTests {
    @Test
    fun testAddition() {
        assertEquals(4, 2 + 2)
    }
}

Escribiendo Pruebas Unitarias Efectivas

  • Aislar Pruebas: Asegúrate de que cada prueba sea independiente.
  • Usar Aserciones: Valida el resultado esperado.
  • Nombrar Pruebas Claramente: Usa nombres descriptivos para los métodos de prueba.

Ejemplo

import org.junit.jupiter.api.Assertions.*
import org.junit.jupiter.api.Test

class CalculatorTests {
    @Test
    fun `addition of two positive numbers`() {
        val result = 2 + 3
        assertEquals(5, result)
    }

    @Test
    fun `subtraction of two numbers`() {
        val result = 5 - 3
        assertEquals(2, result)
    }
}

Simulación de Dependencias

Cuando pruebes clases que tienen dependencias, usa marcos de simulación como Mockito para simular el comportamiento de esas dependencias.

Ejemplo con Mockito

  1. Agrega la dependencia de Mockito:

    dependencies {
    testImplementation("org.mockito:mockito-core:3.7.7")
}

  2. Simula una dependencia:

    import org.junit.jupiter.api.Test
import org.mockito.Mockito.*

class ServiceTests {
    @Test
    fun `test service with mocked repository`() {
        val repository = mock(MyRepository::class.java)
        `when`(repository.getData()).thenReturn("Mocked Data")

        val service = MyService(repository)
        val result = service.fetchData()

        assertEquals("Mocked Data", result)
    }
}

Pruebas Parametrizadas

JUnit 5 soporta pruebas parametrizadas, permitiéndote ejecutar la misma prueba con diferentes entradas.

Ejemplo

import org.junit.jupiter.params.ParameterizedTest
import org.junit.jupiter.params.provider.ValueSource
import org.junit.jupiter.api.Assertions.*

class ParameterizedTests {
    @ParameterizedTest
    @ValueSource(strings = ["racecar", "radar", "level"])
    fun `test palindromes`(word: String) {
        assertTrue(word == word.reversed())
    }
}

Cobertura de Pruebas

Asegúrate de que tus pruebas cubran la mayor parte posible de tu código. Herramientas como JaCoCo pueden ayudar a medir la cobertura de las pruebas.

Ejemplo con JaCoCo

  1. Agrega el plugin JaCoCo a build.gradle.kts:

    plugins {
    id("jacoco")
}

  2. Genera el informe de cobertura:

    ./gradlew test jacocoTestReport

Integración Continua

Integra tus pruebas en una tubería CI/CD para ejecutar automáticamente las pruebas en cada commit. Las herramientas de CI populares incluyen Jenkins, Travis CI y GitHub Actions.

Ejemplo de Flujo de Trabajo de GitHub Actions

name: Kotlin CI

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
    - uses: actions/checkout@v2
    - name: Set up JDK 11
      uses: actions/setup-java@v1
      with:
        java-version: 11
    - name: Build with Gradle
      run: ./gradlew build
    - name: Run tests
      run: ./gradlew test

Conclusión

Las pruebas son una parte esencial del desarrollo en Kotlin. Siguiendo las mejores prácticas como aislar pruebas, usar aserciones, simular dependencias e integrar pruebas en tuberías CI/CD, puedes asegurar que tu código sea confiable y mantenible. Recuerda mantener tus pruebas claras, concisas y comprensivas para cubrir la mayor cantidad de escenarios posible.
