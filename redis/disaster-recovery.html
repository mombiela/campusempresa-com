Recuperación ante Desastres en Redis
Actualizando Redis ►

Introducción

La recuperación ante desastres es un aspecto crítico de la gestión de cualquier sistema de bases de datos, incluyendo Redis. Implica prepararse y recuperarse de eventos inesperados que pueden causar pérdida de datos o tiempo de inactividad. Esta sección te guiará a través de los conceptos esenciales, estrategias y herramientas para implementar la recuperación ante desastres en Redis.

Conceptos Clave

  • Persistencia de Datos: Asegurarse de que los datos se guarden en disco para prevenir pérdidas en caso de un fallo.
  • Replicación: Crear copias de tus datos de Redis en múltiples servidores para asegurar alta disponibilidad.
  • Copia de Seguridad y Restauración: Guardar regularmente instantáneas de tus datos y tener un plan para restaurarlas.
  • Conmutación por Error: Cambiar automáticamente a un servidor de reserva si el servidor principal falla.

Persistencia de Datos

Redis proporciona dos mecanismos principales para la persistencia de datos:

  1. RDB (Copia de Seguridad de la Base de Datos Redis):

    • Crea instantáneas de tu conjunto de datos en puntos de tiempo específicos.
    • Adecuado para copias de seguridad poco frecuentes y reinicios más rápidos.
    • Ejemplo de configuración: 
      save 900 1
save 300 10
save 60 10000
      Esta configuración guarda una instantánea si:
      • Al menos 1 clave cambió en 900 segundos (15 minutos).
      • Al menos 10 claves cambiaron en 300 segundos (5 minutos).
      • Al menos 10000 claves cambiaron en 60 segundos (1 minuto).

  2. AOF (Archivo de Solo Adición):

    • Registra cada operación de escritura recibida por el servidor.
    • Más duradero pero puede ser más lento que RDB.
    • Ejemplo de configuración: 
      appendonly yes
appendfsync everysec
      Esta configuración habilita AOF y lo configura para sincronizarse con el disco cada segundo.

Replicación

La replicación en Redis te permite crear réplicas de tu servidor Redis principal. Esto es crucial para la alta disponibilidad y la recuperación ante desastres.

  • Configuración de una réplica:

    replicaof <master-ip> <master-port>

    Ejemplo:

    replicaof 127.0.0.1 6379

  • Promoción de una réplica a maestro: En caso de que el maestro falle, puedes promover una réplica a maestro:

    SLAVEOF NO ONE

Copia de Seguridad y Restauración

Las copias de seguridad regulares son esenciales para la recuperación ante desastres. Redis soporta la creación de copias de seguridad usando instantáneas RDB.

  • Creación de una copia de seguridad: Puedes activar manualmente una instantánea:

    BGSAVE

    Este comando crea un archivo RDB en el directorio de datos de Redis.

  • Restauración desde una copia de seguridad: Para restaurar, detén el servidor Redis, reemplaza el archivo de volcado existente con tu copia de seguridad y reinicia el servidor:

    cp /path/to/backup/dump.rdb /var/lib/redis/dump.rdb
sudo service redis-server restart

Conmutación por Error

La conmutación por error es el proceso de cambiar a un servidor de reserva cuando el servidor principal falla. Redis Sentinel es una herramienta que proporciona alta disponibilidad y monitoreo.

  • Configuración de Redis Sentinel: Crea un archivo de configuración de sentinel (sentinel.conf):

    sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
sentinel parallel-syncs mymaster 1
    • mymaster: Nombre del maestro.
    • 127.0.0.1: Dirección IP del maestro.
    • 6379: Puerto del maestro.
    • 2: Número de sentinels que necesitan estar de acuerdo antes de iniciar una conmutación por error.

  • Inicio de Redis Sentinel:

    redis-sentinel /path/to/sentinel.conf

Conclusión

La recuperación ante desastres en Redis implica una combinación de persistencia de datos, replicación, copias de seguridad regulares y mecanismos de conmutación por error. Al entender e implementar estas estrategias, puedes asegurar que tu despliegue de Redis sea resiliente y pueda recuperarse rápidamente de eventos inesperados. Prueba regularmente tu plan de recuperación ante desastres para asegurarte de que funcione como se espera y mantén tus configuraciones actualizadas con tu infraestructura en evolución.
