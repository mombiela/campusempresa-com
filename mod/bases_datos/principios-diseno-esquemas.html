Introducción

El diseño de esquemas es una parte fundamental en la creación de bases de datos eficientes y escalables. Un esquema bien diseñado facilita la gestión de datos, mejora el rendimiento de las consultas y asegura la integridad de la información. En esta sección, exploraremos los principios clave del diseño de esquemas, proporcionando ejemplos y ejercicios para reforzar los conceptos.

Conceptos Clave

  • Normalización: Proceso de estructurar una base de datos para reducir la redundancia y mejorar la integridad de los datos.
  • Desnormalización: Proceso de combinar tablas para mejorar el rendimiento de las consultas a costa de la redundancia.
  • Integridad Referencial: Garantiza que las relaciones entre tablas permanezcan consistentes.
  • Cardinalidad: Define la naturaleza de las relaciones entre entidades (uno a uno, uno a muchos, muchos a muchos).
  • Claves Primarias y Foráneas: Identificadores únicos para registros y referencias entre tablas.

Normalización

La normalización es el proceso de organizar los datos en una base de datos para reducir la redundancia y mejorar la integridad de los datos. Existen varias formas normales (NF) que se aplican en este proceso:

  • Primera Forma Normal (1NF): Eliminar grupos repetitivos y asegurar que cada campo contiene solo un valor atómico.
  • Segunda Forma Normal (2NF): Asegurar que todos los atributos no clave dependan completamente de la clave primaria.
  • Tercera Forma Normal (3NF): Eliminar dependencias transitivas, asegurando que los atributos no clave dependan solo de la clave primaria.

Ejemplo de Normalización

Supongamos que tenemos una tabla de empleados con la siguiente estructura:

| EmployeeID | Name | Department | DepartmentHead | |------------|-------|------------|----------------| | 1 | Alice | HR | Bob | | 2 | John | IT | Carol | | 3 | Mike | HR | Bob |

Para normalizar esta tabla a 2NF, podemos dividirla en dos tablas:

Tabla de Empleados:

| EmployeeID | Name | DepartmentID | |------------|-------|--------------| | 1 | Alice | 1 | | 2 | John | 2 | | 3 | Mike | 1 |

Tabla de Departamentos:

| DepartmentID | Department | DepartmentHead | |--------------|------------|----------------| | 1 | HR | Bob | | 2 | IT | Carol |

Desnormalización

La desnormalización es el proceso de combinar tablas para mejorar el rendimiento de las consultas, a costa de introducir redundancia. Es útil en escenarios donde las consultas de lectura son más frecuentes que las de escritura.

Ejemplo de Desnormalización

Si las consultas frecuentes requieren información tanto de empleados como de departamentos, podríamos desnormalizar las tablas anteriores en una sola:

| EmployeeID | Name | Department | DepartmentHead | |------------|-------|------------|----------------| | 1 | Alice | HR | Bob | | 2 | John | IT | Carol | | 3 | Mike | HR | Bob |

Integridad Referencial

La integridad referencial asegura que las relaciones entre tablas permanezcan consistentes. Esto se logra mediante el uso de claves foráneas.

Ejemplo de Integridad Referencial

En la tabla de empleados, DepartmentID es una clave foránea que referencia DepartmentID en la tabla de departamentos. Esto asegura que cada empleado esté asociado con un departamento válido.

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50),
    DepartmentHead VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Cardinalidad

La cardinalidad define la naturaleza de las relaciones entre entidades. Las relaciones pueden ser:

  • Uno a Uno (1:1): Un registro en una tabla está relacionado con un solo registro en otra tabla.
  • Uno a Muchos (1:N): Un registro en una tabla está relacionado con múltiples registros en otra tabla.
  • Muchos a Muchos (M:N): Múltiples registros en una tabla están relacionados con múltiples registros en otra tabla.

Ejemplo de Cardinalidad

Uno a Muchos (1:N): Un departamento tiene muchos empleados.

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Claves Primarias y Foráneas

  • Clave Primaria: Un identificador único para cada registro en una tabla.
  • Clave Foránea: Un campo en una tabla que es una clave primaria en otra tabla, utilizado para establecer y reforzar un vínculo entre los datos de las dos tablas.

Ejemplo de Claves Primarias y Foráneas

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Conclusión

El diseño de esquemas es crucial para la eficiencia y la integridad de una base de datos. La normalización y desnormalización son técnicas complementarias que deben ser aplicadas según las necesidades específicas del sistema. La integridad referencial y la correcta definición de cardinalidad aseguran que los datos permanezcan consistentes y precisos. Comprender y aplicar estos principios es esencial para cualquier profesional de TI que trabaje con bases de datos.
