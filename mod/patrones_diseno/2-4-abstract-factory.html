Módulo 2: Patrones Creacionales

Abstract Factory

¿Qué es el Patrón Abstract Factory?

El patrón Abstract Factory es un patrón de diseño creacional que proporciona una interfaz para crear familias de objetos relacionados o dependientes sin especificar sus clases concretas. Este patrón es útil cuando el sistema debe ser independiente de cómo se crean, componen y representan sus productos.

Conceptos Clave

  • Abstract Factory: Declara una interfaz para operaciones que crean productos abstractos.
  • Concrete Factory: Implementa las operaciones para crear productos concretos.
  • Abstract Product: Declara una interfaz para un tipo de producto objeto.
  • Concrete Product: Define un objeto de producto que implementa la interfaz de producto abstracto.
  • Client: Usa solo las interfaces declaradas por las fábricas abstractas y los productos abstractos.

Ejemplo en Código

Supongamos que estamos desarrollando una aplicación que puede crear diferentes tipos de botones y checkboxes para diferentes sistemas operativos (Windows y Mac).

Paso 1: Definir las Interfaces Abstractas

// Abstract Product A
interface Button {
    void paint();
}

// Abstract Product B
interface Checkbox {
    void paint();
}

Paso 2: Implementar los Productos Concretos

// Concrete Product A1
class WindowsButton implements Button {
    public void paint() {
        System.out.println("Render a button in Windows style.");
    }
}

// Concrete Product A2
class MacButton implements Button {
    public void paint() {
        System.out.println("Render a button in Mac style.");
    }
}

// Concrete Product B1
class WindowsCheckbox implements Checkbox {
    public void paint() {
        System.out.println("Render a checkbox in Windows style.");
    }
}

// Concrete Product B2
class MacCheckbox implements Checkbox {
    public void paint() {
        System.out.println("Render a checkbox in Mac style.");
    }
}

Paso 3: Definir la Fábrica Abstracta

// Abstract Factory
interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

Paso 4: Implementar las Fábricas Concretas

// Concrete Factory 1
class WindowsFactory implements GUIFactory {
    public Button createButton() {
        return new WindowsButton();
    }

    public Checkbox createCheckbox() {
        return new WindowsCheckbox();
    }
}

// Concrete Factory 2
class MacFactory implements GUIFactory {
    public Button createButton() {
        return new MacButton();
    }

    public Checkbox createCheckbox() {
        return new MacCheckbox();
    }
}

Paso 5: Usar la Fábrica en el Cliente

class Application {
    private Button button;
    private Checkbox checkbox;

    public Application(GUIFactory factory) {
        button = factory.createButton();
        checkbox = factory.createCheckbox();
    }

    public void paint() {
        button.paint();
        checkbox.paint();
    }
}

public class Main {
    public static void main(String[] args) {
        GUIFactory factory;
        String osName = System.getProperty("os.name").toLowerCase();
        if (osName.contains("win")) {
            factory = new WindowsFactory();
        } else {
            factory = new MacFactory();
        }

        Application app = new Application(factory);
        app.paint();
    }
}

Ventajas y Desventajas

Ventajas

  • Consistencia de Productos: Garantiza que los productos creados sean compatibles entre sí.
  • Facilidad de Intercambio: Permite cambiar fácilmente la familia de productos que se utilizan en el sistema.
  • Aislamiento de Código: Aísla el código cliente de las clases concretas de productos.

Desventajas

  • Complejidad: Puede añadir complejidad al sistema debido a la creación de múltiples interfaces y clases.
  • Dificultad de Extensión: Añadir nuevas familias de productos puede requerir cambios significativos en el código existente.

Conclusión

El patrón Abstract Factory es una poderosa herramienta para crear familias de objetos relacionados sin acoplar el código cliente a las clases concretas. Aunque puede añadir complejidad, su uso es beneficioso en sistemas que requieren consistencia y flexibilidad en la creación de objetos. Este patrón es ampliamente utilizado en aplicaciones que necesitan ser portables entre diferentes plataformas o que requieren una arquitectura modular y extensible.
