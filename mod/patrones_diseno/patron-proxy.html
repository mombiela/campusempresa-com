Introducción al Patrón Proxy

El patrón Proxy es un patrón de diseño estructural que proporciona un sustituto o marcador de posición para otro objeto. Un proxy controla el acceso a un objeto, permitiendo realizar ciertas acciones antes o después de la solicitud al objeto real.

Objetivos del Patrón Proxy

  • Control de acceso: Restringir el acceso a ciertos métodos o datos.
  • Optimización: Cargar objetos pesados solo cuando se necesitan.
  • Seguridad: Proteger el objeto real de accesos no autorizados.
  • Registro y monitoreo: Registrar las solicitudes y respuestas para auditoría o monitoreo.

Tipos de Proxy

  • Proxy Virtual: Maneja la creación y el acceso a objetos costosos en términos de recursos.
  • Proxy de Protección: Controla el acceso a un objeto, permitiendo diferentes permisos para diferentes usuarios.
  • Proxy Remoto: Representa un objeto que reside en un espacio de direcciones diferente (por ejemplo, en una máquina remota).
  • Proxy de Caché: Almacena los resultados de operaciones costosas para evitar repetirlas.

Estructura del Patrón Proxy

El patrón Proxy generalmente involucra los siguientes componentes:

  • Subject (Sujeto): Define la interfaz común para el RealSubject y el Proxy.
  • RealSubject (Sujeto Real): El objeto real que el proxy representa.
  • Proxy: Mantiene una referencia al RealSubject y controla el acceso a él.

Diagrama UML

| Componente | Descripción | |--------------|-----------------------------------------------------------------------------| | Subject | Interfaz común para RealSubject y Proxy. | | RealSubject | El objeto real que se está representando. | | Proxy | Controla el acceso al RealSubject y puede añadir funcionalidad adicional. |

Ejemplo de Código

A continuación, se presenta un ejemplo en Java para ilustrar el patrón Proxy:

// Interfaz común
public interface Image {
    void display();
}

// Sujeto Real
public class RealImage implements Image {
    private String fileName;

    public RealImage(String fileName) {
        this.fileName = fileName;
        loadFromDisk();
    }

    private void loadFromDisk() {
        System.out.println("Loading " + fileName);
    }

    @Override
    public void display() {
        System.out.println("Displaying " + fileName);
    }
}

// Proxy
public class ProxyImage implements Image {
    private RealImage realImage;
    private String fileName;

    public ProxyImage(String fileName) {
        this.fileName = fileName;
    }

    @Override
    public void display() {
        if (realImage == null) {
            realImage = new RealImage(fileName);
        }
        realImage.display();
    }
}

// Cliente
public class ProxyPatternDemo {
    public static void main(String[] args) {
        Image image = new ProxyImage("test_image.jpg");

        // La imagen se cargará desde el disco
        image.display();
        System.out.println("");

        // La imagen no se cargará desde el disco
        image.display();
    }
}

Explicación del Código

  • Interfaz Image: Define el método display() que será implementado por RealImage y ProxyImage.
  • RealImage: Implementa la interfaz Image y representa el objeto real que se carga desde el disco.
  • ProxyImage: Implementa la interfaz Image y controla el acceso a RealImage. Solo carga RealImage cuando es necesario.
  • ProxyPatternDemo: Clase cliente que utiliza ProxyImage para acceder a RealImage.

Ventajas y Desventajas

Ventajas

  • Control de acceso: Permite controlar el acceso al objeto real.
  • Optimización de recursos: Carga objetos pesados solo cuando son necesarios.
  • Seguridad: Puede añadir capas de seguridad adicionales.
  • Monitoreo y registro: Facilita el registro de las solicitudes y respuestas.

Desventajas

  • Complejidad adicional: Añade complejidad al sistema debido a la introducción de nuevas clases.
  • Rendimiento: Puede introducir una ligera sobrecarga debido a la delegación de solicitudes.

Conclusión

El patrón Proxy es una herramienta poderosa para controlar el acceso a objetos, optimizar el uso de recursos y añadir capas de seguridad y monitoreo. Sin embargo, debe ser utilizado con cuidado para evitar añadir complejidad innecesaria al sistema. Al comprender y aplicar correctamente este patrón, los desarrolladores pueden crear sistemas más eficientes y seguros.
