Políticas de Seguridad de Pods
Definiciones de Recursos Personalizados (CRDs) ►

Las Políticas de Seguridad de Pods (PSPs) son un aspecto crítico de la seguridad de Kubernetes, proporcionando un mecanismo para controlar los aspectos relacionados con la seguridad de la creación y actualización de pods. Este tema te guiará desde los conceptos básicos hasta los avanzados de las PSPs, asegurando una comprensión completa.

Introducción a las Políticas de Seguridad de Pods

Las Políticas de Seguridad de Pods son recursos a nivel de clúster que controlan aspectos sensibles a la seguridad de la especificación de los pods. Permiten a los administradores definir un conjunto de condiciones que un pod debe cumplir para ser aceptado en el sistema.

Conceptos Clave

  • Política de Seguridad de Pods (PSP): Un recurso a nivel de clúster que define un conjunto de condiciones que un pod debe cumplir para ser aceptado en el sistema.
  • Controlador de Admisión: Un fragmento de código que intercepta las solicitudes al servidor API de Kubernetes antes de la persistencia del objeto, pero después de que la solicitud sea autenticada y autorizada.
  • Contexto de Seguridad: Define configuraciones de privilegios y control de acceso para un Pod o Contenedor.

Creando una Política de Seguridad de Pods

Para crear una PSP, necesitas definir un archivo de configuración YAML que especifique los requisitos de seguridad para tus pods.

Ejemplo de YAML de PSP

apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: example-psp
spec:
  privileged: false
  allowPrivilegeEscalation: false
  requiredDropCapabilities:
    - ALL
  volumes:
    - 'configMap'
    - 'emptyDir'
    - 'projected'
    - 'secret'
  hostNetwork: false
  hostIPC: false
  hostPID: false
  runAsUser:
    rule: 'MustRunAsNonRoot'
  seLinux:
    rule: 'RunAsAny'
  supplementalGroups:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
  fsGroup:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535

Explicación

  • privileged: Asegura que no se permitan contenedores privilegiados.
  • allowPrivilegeEscalation: Previene la escalada de privilegios.
  • requiredDropCapabilities: Elimina todas las capacidades para minimizar la superficie de ataque potencial.
  • volumes: Restringe los tipos de volúmenes que se pueden usar.
  • hostNetwork, hostIPC, hostPID: Previene el uso de las redes, IPC y espacios de nombres PID del host.
  • runAsUser: Asegura que los contenedores se ejecuten como un usuario no root.
  • seLinux: Permite cualquier contexto SELinux.
  • supplementalGroups: Requiere que los contenedores se ejecuten con grupos suplementarios específicos.
  • fsGroup: Requiere que los contenedores se ejecuten con grupos de sistemas de archivos específicos.

Aplicando una Política de Seguridad de Pods

Para aplicar una PSP, necesitas habilitar el controlador de admisión de PodSecurityPolicy y crear los RoleBindings o ClusterRoleBindings apropiados.

Habilitando el Controlador de Admisión

Edita la configuración del servidor API de Kubernetes para incluir el controlador de admisión PodSecurityPolicy.

--enable-admission-plugins=PodSecurityPolicy

Creando RoleBindings

Necesitas crear RoleBindings o ClusterRoleBindings para vincular la PSP a usuarios específicos o cuentas de servicio.

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: use-example-psp
  namespace: default
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: use-example-psp
subjects:
- kind: User
  name: example-user
  apiGroup: rbac.authorization.k8s.io

Explicación

  • roleRef: Hace referencia al rol que otorga permisos para usar la PSP.
  • subjects: Especifica los usuarios o cuentas de servicio a los que se aplica el RoleBinding.

Uso Avanzado de las Políticas de Seguridad de Pods

Combinando Múltiples PSPs

Puedes crear múltiples PSPs para atender diferentes requisitos de seguridad para diferentes espacios de nombres o aplicaciones.

Ejemplo

apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted-psp
spec:
  privileged: false
  allowPrivilegeEscalation: false
  requiredDropCapabilities:
    - ALL
  runAsUser:
    rule: 'MustRunAsNonRoot'
  seLinux:
    rule: 'RunAsAny'
  supplementalGroups:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
  fsGroup:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: permissive-psp
spec:
  privileged: true
  allowPrivilegeEscalation: true
  runAsUser:
    rule: 'RunAsAny'
  seLinux:
    rule: 'RunAsAny'
  supplementalGroups:
    rule: 'RunAsAny'
  fsGroup:
    rule: 'RunAsAny'

Explicación

  • restricted-psp: Impone políticas de seguridad estrictas.
  • permissive-psp: Permite configuraciones más permisivas para casos de uso específicos.

Conclusión

Las Políticas de Seguridad de Pods son una herramienta poderosa para imponer estándares de seguridad en un clúster de Kubernetes. Al comprender e implementar las PSPs, puedes mejorar significativamente la postura de seguridad de tu entorno de Kubernetes. Recuerda:

  • Definir PSPs claras y concisas adaptadas a tus requisitos de seguridad.
  • Aplicar PSPs usando RoleBindings o ClusterRoleBindings.
  • Revisar y actualizar regularmente las PSPs para adaptarse a las necesidades de seguridad en evolución.

Al dominar las PSPs, aseguras que tus clústeres de Kubernetes sean seguros y cumplan con las mejores prácticas.
