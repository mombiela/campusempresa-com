Introducción a WSDL y UDDI

En el contexto de la Arquitectura Orientada a Servicios (SOA), WSDL (Web Services Description Language) y UDDI (Universal Description, Discovery, and Integration) son componentes fundamentales para la descripción y descubrimiento de servicios web. Este tema cubre los conceptos esenciales, la estructura, y el uso de WSDL y UDDI, proporcionando ejemplos claros y ejercicios prácticos.

WSDL (Web Services Description Language)

WSDL es un lenguaje basado en XML que se utiliza para describir los servicios web y cómo acceder a ellos. Proporciona una forma estándar de describir los detalles de los servicios web, incluyendo los métodos disponibles, los parámetros requeridos y los tipos de datos utilizados.

Conceptos Clave de WSDL

  • Definición del Servicio: Describe el servicio web en su totalidad.
  • Tipos: Define los tipos de datos utilizados por el servicio.
  • Mensajes: Describe los datos que se intercambian entre el cliente y el servicio.
  • Operaciones: Define las acciones que se pueden realizar en el servicio.
  • Enlaces (Bindings): Describe el protocolo y el formato de datos para cada operación.
  • Puerto (Port): Define una dirección de red para el servicio.

Estructura de un Documento WSDL

Un documento WSDL típico incluye las siguientes secciones:

  1. : Define los tipos de datos complejos utilizados por el servicio.
  2. : Define los mensajes que se intercambian.
  3. : Define las operaciones del servicio.
  4. : Define el protocolo y el formato de datos.
  5. : Define la URL del servicio.

Ejemplo de Documento WSDL

<definitions xmlns="http://schemas.xmlsoap.org/wsdl/" 
             xmlns:soap="http://schemas.xmlsoap.org/wsdl/soap/" 
             xmlns:tns="http://example.com/wsdl" 
             targetNamespace="http://example.com/wsdl">
  <types>
    <xsd:schema targetNamespace="http://example.com/wsdl">
      <xsd:element name="getGreeting" type="xsd:string"/>
      <xsd:element name="getGreetingResponse" type="xsd:string"/>
    </xsd:schema>
  </types>
  <message name="getGreetingRequest">
    <part name="name" element="tns:getGreeting"/>
  </message>
  <message name="getGreetingResponse">
    <part name="greeting" element="tns:getGreetingResponse"/>
  </message>
  <portType name="GreetingPortType">
    <operation name="getGreeting">
      <input message="tns:getGreetingRequest"/>
      <output message="tns:getGreetingResponse"/>
    </operation>
  </portType>
  <binding name="GreetingBinding" type="tns:GreetingPortType">
    <soap:binding style="rpc" transport="http://schemas.xmlsoap.org/soap/http"/>
    <operation name="getGreeting">
      <soap:operation soapAction="getGreeting"/>
      <input>
        <soap:body use="literal"/>
      </input>
      <output>
        <soap:body use="literal"/>
      </output>
    </operation>
  </binding>
  <service name="GreetingService">
    <port name="GreetingPort" binding="tns:GreetingBinding">
      <soap:address location="http://example.com/greeting"/>
    </port>
  </service>
</definitions>

UDDI (Universal Description, Discovery, and Integration)

UDDI es un estándar para la publicación y descubrimiento de servicios web. Actúa como una "página amarilla" para los servicios web, permitiendo a las empresas registrar y buscar servicios disponibles.

Conceptos Clave de UDDI

  • Registro de Negocios: Información sobre la empresa que ofrece el servicio.
  • Registro de Servicios: Información sobre los servicios web que ofrece la empresa.
  • Modelos de Negocio: Descripciones de las categorías y clasificaciones de los servicios.
  • Interfaz de Programación de Aplicaciones (API): Métodos para interactuar con el registro UDDI.

Estructura de UDDI

UDDI utiliza una estructura jerárquica para organizar la información:

  • BusinessEntity: Representa una empresa.
  • BusinessService: Representa un servicio ofrecido por la empresa.
  • BindingTemplate: Detalles técnicos sobre cómo acceder al servicio.
  • tModel: Modelos técnicos que describen las especificaciones del servicio.

Ejemplo de Registro UDDI

<businessEntity businessKey="uuid:12345">
  <name>Example Company</name>
  <description>Provider of web services</description>
  <businessServices>
    <businessService serviceKey="uuid:67890">
      <name>Greeting Service</name>
      <bindingTemplates>
        <bindingTemplate bindingKey="uuid:abcde">
          <accessPoint URLType="http">http://example.com/greeting</accessPoint>
        </bindingTemplate>
      </bindingTemplates>
    </businessService>
  </businessServices>
</businessEntity>

Comparación entre WSDL y UDDI

| Característica | WSDL | UDDI | |-------------------------|-------------------------------------------|-------------------------------------------| | Propósito | Describir servicios web | Registrar y descubrir servicios web | | Formato | XML | XML | | Componentes Principales | Definición del servicio, tipos, mensajes | BusinessEntity, BusinessService, tModel | | Uso | Especificación técnica del servicio | Registro y búsqueda de servicios |

Conclusión

WSDL y UDDI son componentes esenciales en la Arquitectura Orientada a Servicios (SOA). Mientras que WSDL proporciona una descripción detallada y técnica de los servicios web, UDDI facilita el registro y descubrimiento de estos servicios. Comprender ambos estándares es crucial para diseñar, implementar y gestionar servicios web eficientemente.

Ejercicio Práctico

  1. Crear un Documento WSDL: Escriba un documento WSDL para un servicio web que proporcione información meteorológica.
  2. Registrar un Servicio en UDDI: Cree un registro UDDI para el servicio meteorológico descrito en el ejercicio anterior.

Estos ejercicios ayudarán a consolidar el conocimiento adquirido y a aplicar los conceptos en escenarios prácticos.
