Introducción

La replicación y la alta disponibilidad son conceptos críticos en la gestión de bases de datos, asegurando que tu base de datos PostgreSQL permanezca accesible y resiliente frente a fallos de hardware, errores de software u otras interrupciones. Esta sección te guiará a través de los conceptos básicos de la replicación, los diferentes métodos de replicación y las estrategias para lograr alta disponibilidad en PostgreSQL.

Conceptos Clave

  • Replicación: El proceso de copiar datos de un servidor de base de datos (primario) a otro (réplica).
  • Alta Disponibilidad (HA): Asegurar que el sistema de base de datos esté continuamente operativo con un tiempo de inactividad mínimo.
  • Servidor Primario: El servidor principal donde ocurren todas las operaciones de escritura.
  • Servidor Réplica: Un servidor que mantiene una copia de los datos del servidor primario, generalmente para operaciones de lectura o propósitos de conmutación por error.
  • Conmutación por Error: El proceso de cambiar a un servidor réplica cuando el servidor primario falla.

Tipos de Replicación

PostgreSQL soporta varios métodos de replicación, cada uno con sus propios casos de uso y configuraciones.

Replicación en Streaming

La replicación en streaming es un método donde el servidor primario envía continuamente cambios al servidor réplica en tiempo real.

  • Ventajas:

    • Replicación de datos en tiempo real.
    • Baja latencia.
    • Adecuado para configuraciones de alta disponibilidad.

  • Desventajas:

    • Requiere una conexión de red estable.
    • La réplica puede estar ligeramente desfasada respecto al servidor primario.

Ejemplo de Configuración

  1. Configuración del Servidor Primario:

    # postgresql.conf
wal_level = replica
max_wal_senders = 3
wal_keep_segments = 64
hot_standby = on

# pg_hba.conf
host replication replicator 192.168.1.0/24 md5

  2. Configuración del Servidor Réplica:

    pg_basebackup -h primary_ip -D /var/lib/postgresql/12/main -U replicator -P -v --wal-method=stream

    # recovery.conf
standby_mode = 'on'
primary_conninfo = 'host=primary_ip port=5432 user=replicator password=yourpassword'
trigger_file = '/tmp/trigger'

Replicación Lógica

La replicación lógica permite más flexibilidad al replicar cambios de datos basados en cambios lógicos en lugar de cambios físicos.

  • Ventajas:

    • Permite la replicación selectiva de tablas.
    • Puede replicar entre diferentes versiones de PostgreSQL.
    • Adecuado para procesos de almacenamiento de datos y ETL.

  • Desventajas:

    • Configuración más compleja.
    • Potencialmente mayor latencia en comparación con la replicación en streaming.

Ejemplo de Configuración

  1. Configuración del Servidor Primario:

    # postgresql.conf
wal_level = logical
max_replication_slots = 4
max_wal_senders = 4

# pg_hba.conf
host replication replicator 192.168.1.0/24 md5

  2. Creación de una Publicación:

    CREATE PUBLICATION my_publication FOR TABLE my_table;

  3. Configuración del Servidor Réplica:

    CREATE SUBSCRIPTION my_subscription CONNECTION 'host=primary_ip dbname=mydb user=replicator password=yourpassword' PUBLICATION my_publication;

Estrategias de Alta Disponibilidad

La alta disponibilidad en PostgreSQL implica más que solo replicación. Incluye monitoreo, conmutación por error automática y balanceo de carga.

Monitoreo

Las herramientas de monitoreo ayudan a detectar fallos y problemas de rendimiento. Las herramientas comunes incluyen:

  • pg_stat_replication: Proporciona información sobre el estado de la replicación.
  • pg_stat_activity: Muestra la actividad actual de la base de datos.
  • Herramientas de terceros: Herramientas como Nagios, Zabbix y Prometheus pueden ser utilizadas para un monitoreo integral.

Conmutación por Error Automática

La conmutación por error automática asegura que un servidor réplica tome el control cuando el servidor primario falla.

  • Herramientas:
    • Patroni: Una solución de alta disponibilidad para PostgreSQL.
    • repmgr: Un conjunto de herramientas de código abierto para gestionar la replicación y la conmutación por error.

Ejemplo con Patroni

  1. Instalar Patroni:

    pip install patroni[etcd]

  2. Archivo de Configuración (patroni.yml):

    scope: my_cluster
namespace: /db/
name: postgresql0

etcd:
  host: 127.0.0.1:2379

postgresql:
  listen: 127.0.0.1:5432
  connect_address: 127.0.0.1:5432
  data_dir: /var/lib/postgresql/12/main
  config_dir: /etc/postgresql/12/main
  bin_dir: /usr/lib/postgresql/12/bin
  authentication:
    replication:
      username: replicator
      password: yourpassword
    superuser:
      username: postgres
      password: yourpassword

Balanceo de Carga

El balanceo de carga distribuye las consultas de lectura entre múltiples servidores réplica para mejorar el rendimiento.

  • pgpool-II: Un middleware que proporciona agrupación de conexiones, balanceo de carga y gestión de replicación.

Ejemplo con pgpool-II

  1. Instalar pgpool-II:

    sudo apt-get install pgpool2

  2. Archivo de Configuración (pgpool.conf):

    listen_addresses = '*'
port = 9999
backend_hostname0 = 'primary_ip'
backend_port0 = 5432
backend_weight0 = 1
backend_data_directory0 = '/var/lib/postgresql/12/main'
backend_flag0 = 'ALLOW_TO_FAILOVER'

backend_hostname1 = 'replica_ip'
backend_port1 = 5432
backend_weight1 = 1
backend_data_directory1 = '/var/lib/postgresql/12/main'
backend_flag1 = 'ALLOW_TO_FAILOVER'

Conclusión

La replicación y la alta disponibilidad son esenciales para mantener un sistema de base de datos PostgreSQL resiliente y de alto rendimiento. Al comprender e implementar los diferentes métodos de replicación y estrategias de alta disponibilidad, puedes asegurar que tu base de datos permanezca operativa y pueda manejar fallos de manera elegante. Ya sea que elijas la replicación en streaming para la consistencia de datos en tiempo real o la replicación lógica para mayor flexibilidad, combinarlas con mecanismos robustos de monitoreo y conmutación por error te ayudará a lograr un entorno PostgreSQL altamente disponible.
