Introducció

El disseny d'esquemes és una part fonamental en la creació de bases de dades eficients i escalables. Un esquema ben dissenyat facilita la gestió de dades, millora el rendiment de les consultes i assegura la integritat de la informació. En aquesta secció, explorarem els principis clau del disseny d'esquemes, proporcionant exemples i exercicis per reforçar els conceptes.

Conceptes Clau

  • Normalització: Procés d'estructurar una base de dades per reduir la redundància i millorar la integritat de les dades.
  • Desnormalització: Procés de combinar taules per millorar el rendiment de les consultes a costa de la redundància.
  • Integritat Referencial: Garanteix que les relacions entre taules romanguin consistents.
  • Cardinalitat: Defineix la naturalesa de les relacions entre entitats (un a un, un a molts, molts a molts).
  • Claus Primàries i Foranes: Identificadors únics per a registres i referències entre taules.

Normalització

La normalització és el procés d'organitzar les dades en una base de dades per reduir la redundància i millorar la integritat de les dades. Hi ha diverses formes normals (NF) que s'apliquen en aquest procés:

  • Primera Forma Normal (1NF): Eliminar grups repetitius i assegurar que cada camp conté només un valor atòmic.
  • Segona Forma Normal (2NF): Assegurar que tots els atributs no clau depenen completament de la clau primària.
  • Tercera Forma Normal (3NF): Eliminar dependències transitives, assegurant que els atributs no clau depenen només de la clau primària.

Exemple de Normalització

Suposem que tenim una taula d'empleats amb la següent estructura:

| EmployeeID | Name | Department | DepartmentHead | |------------|-------|------------|----------------| | 1 | Alice | HR | Bob | | 2 | John | IT | Carol | | 3 | Mike | HR | Bob |

Per normalitzar aquesta taula a 2NF, podem dividir-la en dues taules:

Taula d'Empleats:

| EmployeeID | Name | DepartmentID | |------------|-------|--------------| | 1 | Alice | 1 | | 2 | John | 2 | | 3 | Mike | 1 |

Taula de Departaments:

| DepartmentID | Department | DepartmentHead | |--------------|------------|----------------| | 1 | HR | Bob | | 2 | IT | Carol |

Desnormalització

La desnormalització és el procés de combinar taules per millorar el rendiment de les consultes, a costa d'introduir redundància. És útil en escenaris on les consultes de lectura són més freqüents que les d'escriptura.

Exemple de Desnormalització

Si les consultes freqüents requereixen informació tant d'empleats com de departaments, podríem desnormalitzar les taules anteriors en una sola:

| EmployeeID | Name | Department | DepartmentHead | |------------|-------|------------|----------------| | 1 | Alice | HR | Bob | | 2 | John | IT | Carol | | 3 | Mike | HR | Bob |

Integritat Referencial

La integritat referencial assegura que les relacions entre taules romanguin consistents. Això s'aconsegueix mitjançant l'ús de claus foranes.

Exemple d'Integritat Referencial

A la taula d'empleats, DepartmentID és una clau forana que referencia DepartmentID a la taula de departaments. Això assegura que cada empleat estigui associat amb un departament vàlid.

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50),
    DepartmentHead VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Cardinalitat

La cardinalitat defineix la naturalesa de les relacions entre entitats. Les relacions poden ser:

  • Un a Un (1:1): Un registre en una taula està relacionat amb un sol registre en una altra taula.
  • Un a Molts (1:N): Un registre en una taula està relacionat amb múltiples registres en una altra taula.
  • Molts a Molts (M:N): Múltiples registres en una taula estan relacionats amb múltiples registres en una altra taula.

Exemple de Cardinalitat

Un a Molts (1:N): Un departament té molts empleats.

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Claus Primàries i Foranes

  • Clau Primària: Un identificador únic per a cada registre en una taula.
  • Clau Forana: Un camp en una taula que és una clau primària en una altra taula, utilitzat per establir i reforçar un vincle entre les dades de les dues taules.

Exemple de Claus Primàries i Foranes

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Conclusió

El disseny d'esquemes és crucial per a l'eficiència i la integritat d'una base de dades. La normalització i desnormalització són tècniques complementàries que han de ser aplicades segons les necessitats específiques del sistema. La integritat referencial i la correcta definició de cardinalitat asseguren que les dades romanguin consistents i precises. Comprendre i aplicar aquests principis és essencial per a qualsevol professional de TI que treballi amb bases de dades.
