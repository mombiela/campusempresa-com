Introducció

En un entorn de microserveis, la comunicació entre serveis és fonamental per al correcte funcionament del sistema. Aquest tema aborda els diferents mètodes i patrons de comunicació entre microserveis, els seus avantatges i desavantatges, i proporciona exemples pràctics per il·lustrar aquests conceptes.

Mètodes de Comunicació

Comunicació Sincrònica

  • HTTP/REST: És el mètode més comú per a la comunicació entre microserveis. Utilitza el protocol HTTP i segueix els principis RESTful.

    • Avantatges:
      • Simplicitat i àmplia adopció.
      • Fàcil d'implementar i depurar.
    • Desavantatges:
      • Latència i sobrecàrrega de xarxa.
      • Dependència directa entre serveis.
    • Exemple de Codi: 
      # Servei A
import requests

response = requests.get('http://servei-b/api/recurs')
data = response.json()

  • gRPC: Un framework de comunicació d'alt rendiment desenvolupat per Google que utilitza HTTP/2 i Protocol Buffers.

    • Avantatges:
      • Alta eficiència i menor latència.
      • Suport per a múltiples llenguatges.
    • Desavantatges:
      • Major complexitat en la configuració.
      • Menor adopció en comparació amb HTTP/REST.
    • Exemple de Codi: 
      # Servei A
import grpc
import servei_b_pb2
import servei_b_pb2_grpc

channel = grpc.insecure_channel('servei-b:50051')
stub = servei_b_pb2_grpc.ServeiBStub(channel)
response = stub.ObtenirRecurs(servei_b_pb2.RecursRequest(id=1))

Comunicació Asincrònica

  • Missatgeria: Utilitza sistemes de missatgeria com RabbitMQ, Kafka o AWS SQS per a la comunicació entre serveis.
    • Avantatges:
      • Desacoblament entre serveis.
      • Millor maneig de la càrrega i resiliència.
    • Desavantatges:
      • Complexitat en la gestió de missatges.
      • Latència en el lliurament de missatges.
    • Exemple de Codi: 
      # Servei A (publicador)
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.queue_declare(queue='cua_servei_b')
channel.basic_publish(exchange='', routing_key='cua_servei_b', body='Missatge per a Servei B')
connection.close()

# Servei B (subscriptor)
import pika

def callback(ch, method, properties, body):
    print(f"Rebut {body}")

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.queue_declare(queue='cua_servei_b')
channel.basic_consume(queue='cua_servei_b', on_message_callback=callback, auto_ack=True)
channel.start_consuming()

Patrons de Comunicació

  • API Gateway: Un punt d'entrada únic per a totes les sol·licituds de clients, que redirigeix les sol·licituds als microserveis corresponents.

    • Avantatges:
      • Simplifica la comunicació amb els clients.
      • Permet la implementació de polítiques de seguretat i autenticació.
    • Desavantatges:
      • Pot convertir-se en un punt únic de fallada.
      • Afegeix latència addicional.

  • Choreography vs Orchestration:

    • Choreography: Cada microservei es comunica directament amb altres serveis necessaris per completar una tasca.
      • Avantatges:
        • Descentralització i menor acoblament.
      • Desavantatges:
        • Difícil de gestionar i depurar.
    • Orchestration: Un servei central (orquestrador) gestiona la comunicació i coordinació entre microserveis.
      • Avantatges:
        • Millor control i visibilitat.
      • Desavantatges:
        • Major acoblament i dependència de l'orquestrador.

Comparació de Mètodes de Comunicació

| Mètode | Avantatges | Desavantatges | |-----------------|---------------------------------------------|--------------------------------------------| | HTTP/REST | Simplicitat, àmplia adopció | Latència, dependència directa | | gRPC | Alta eficiència, suport multi-llenguatge | Complexitat, menor adopció | | Missatgeria | Desacoblament, resiliència | Complexitat en gestió, latència | | API Gateway | Simplificació, seguretat | Punt únic de fallada, latència addicional | | Choreography | Descentralització, menor acoblament | Difícil de gestionar i depurar | | Orchestration | Millor control i visibilitat | Major acoblament, dependència |

Conclusió

La comunicació entre microserveis és un aspecte crític en l'arquitectura de microserveis. L'elecció del mètode de comunicació adequat depèn de diversos factors, incloent la latència, la resiliència, la complexitat i l'acoblament entre serveis. Comprendre els avantatges i desavantatges de cada mètode i patró de comunicació és essencial per dissenyar sistemes robustos i eficients.
