Escalar NGINX és un tema crític per assegurar que les teves aplicacions web puguin gestionar un augment de tràfic i proporcionar alta disponibilitat. Aquesta secció cobrirà els conceptes fonamentals, estratègies i implementacions pràctiques per escalar NGINX des d'un sol servidor fins a una configuració distribuïda i d'alt rendiment.

Conceptes Clau en Escalar NGINX

  • Balanceig de Càrrega: Distribuir el tràfic entrant entre diversos servidors per assegurar que cap servidor esdevingui un coll d'ampolla.
  • Escalat Horitzontal: Afegir més servidors per gestionar una càrrega augmentada.
  • Escalat Vertical: Augmentar els recursos (CPU, RAM) d'un sol servidor.
  • Alta Disponibilitat: Assegurar que el servei romangui disponible fins i tot si un o més servidors fallen.
  • Caché: Emmagatzemar dades accedides freqüentment per reduir la càrrega en els servidors de backend.

Balanceig de Càrrega amb NGINX

El balanceig de càrrega és una estratègia clau per escalar NGINX. Implica distribuir les sol·licituds entrants entre diversos servidors de backend.

Tipus de Balanceig de Càrrega

  • Round Robin: Distribueix les sol·licituds de manera uniforme entre tots els servidors.
  • Least Connections: Envia les sol·licituds al servidor amb menys connexions actives.
  • IP Hash: Distribueix les sol·licituds basant-se en l'adreça IP del client.

Configuració del Balanceig de Càrrega

Aquí tens un exemple de configuració del balanceig de càrrega en NGINX utilitzant el mètode Round Robin:

http {
    upstream backend {
        server backend1.example.com;
        server backend2.example.com;
        server backend3.example.com;
    }

    server {
        listen 80;

        location / {
            proxy_pass http://backend;
        }
    }
}

En aquesta configuració:

  • El bloc upstream defineix un grup de servidors de backend.
  • La directiva proxy_pass redirigeix les sol·licituds entrants al grup backend.

Escalat Horitzontal

L'escalat horitzontal implica afegir més servidors per gestionar una càrrega augmentada. Això es pot aconseguir:

  • Afegint Més Servidors de Backend: Simplement afegeix més servidors al bloc upstream.
  • Utilitzant Serveis al Núvol: Aprofita proveïdors de núvol com AWS, GCP o Azure per escalar dinàmicament el nombre de servidors.

Exemple d'Afegir Més Servidors de Backend

http {
    upstream backend {
        server backend1.example.com;
        server backend2.example.com;
        server backend3.example.com;
        server backend4.example.com;  # Nou servidor afegit
    }

    server {
        listen 80;

        location / {
            proxy_pass http://backend;
        }
    }
}

Alta Disponibilitat

L'alta disponibilitat assegura que el teu servei romangui operatiu fins i tot si alguns servidors fallen. Això es pot aconseguir mitjançant:

  • Comprovacions de Salut: Comprovar regularment la salut dels servidors de backend i eliminar els que no estiguin saludables del grup.
  • Mecanismes de Failover: Redirigir automàticament el tràfic als servidors saludables en cas de fallada.

Configuració de Comprovacions de Salut

NGINX Plus suporta comprovacions de salut actives. Aquí tens un exemple de configuració:

http {
    upstream backend {
        server backend1.example.com;
        server backend2.example.com;
        server backend3.example.com;

        health_check interval=10s fails=3 passes=2;
    }

    server {
        listen 80;

        location / {
            proxy_pass http://backend;
        }
    }
}

En aquesta configuració:

  • La directiva health_check especifica l'interval per a les comprovacions de salut i les condicions per marcar un servidor com a no saludable.

Caché

El caché pot reduir significativament la càrrega en els servidors de backend emmagatzemant dades accedides freqüentment. NGINX suporta diversos mecanismes de caché, incloent:

  • Caché de Proxy: Emmagatzemar respostes dels servidors de backend.
  • Caché de FastCGI: Emmagatzemar respostes dels servidors FastCGI.

Configuració de Caché de Proxy

Aquí tens un exemple de configuració de caché de proxy en NGINX:

http {
    proxy_cache_path /data/nginx/cache levels=1:2 keys_zone=my_cache:10m max_size=10g inactive=60m use_temp_path=off;

    server {
        listen 80;

        location / {
            proxy_cache my_cache;
            proxy_pass http://backend;
            proxy_cache_valid 200 302 10m;
            proxy_cache_valid 404 1m;
        }
    }
}

En aquesta configuració:

  • La directiva proxy_cache_path defineix el camí d'emmagatzematge del caché i els seus paràmetres.
  • La directiva proxy_cache habilita el caché per a la ubicació especificada.

Conclusió

Escalar NGINX implica una combinació de balanceig de càrrega, escalat horitzontal, alta disponibilitat i caché. Entenent i implementant aquestes estratègies, pots assegurar que les teves aplicacions web puguin gestionar un augment de tràfic i romanguin altament disponibles. Comença amb el balanceig de càrrega bàsic i incorpora gradualment tècniques més avançades com les comprovacions de salut i el caché per optimitzar la teva configuració de NGINX.
