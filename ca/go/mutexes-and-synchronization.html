Introducció

En la programació concurrent, gestionar l'accés als recursos compartits és crucial per evitar condicions de carrera i assegurar la consistència de les dades. Go proporciona diversos primitius de sincronització, amb els mutexes sent un dels més fonamentals. Aquesta secció cobrirà els conceptes bàsics dels mutexes, com utilitzar-los en Go i les millors pràctiques per a la sincronització.

Què és un Mutex?

Un mutex, o bloqueig d'exclusió mútua, és un primitiu de sincronització utilitzat per protegir recursos compartits de l'accés concurrent. Quan una goroutine bloqueja un mutex, altres goroutines que intenten bloquejar-lo queden bloquejades fins que el mutex es desbloqueja.

Conceptes Clau

  • Bloqueig: Adquirir el mutex per obtenir accés exclusiu a un recurs.
  • Desbloqueig: Alliberar el mutex per permetre que altres goroutines accedeixin al recurs.
  • Condició de Carrera: Una situació on el comportament del programari depèn del temps relatiu dels esdeveniments, com l'ordre d'execució de les goroutines.

Utilitzant Mutexes en Go

El paquet sync de Go proporciona el tipus Mutex per a l'exclusió mútua. A continuació es mostren els passos per utilitzar un mutex en Go.

Importar el paquet sync

Primer, cal importar el paquet sync:

import "sync"

Declarar un Mutex

Pots declarar un mutex com a part d'una estructura o com una variable independent:

var mu sync.Mutex

Bloquejar i Desbloquejar un Mutex

Per protegir una secció crítica, bloqueges el mutex abans d'accedir al recurs compartit i el desbloqueges després:

mu.Lock()
// Secció crítica: accedir al recurs compartit
mu.Unlock()

Exemple: Protegir un Comptador

Vegem un exemple senzill on múltiples goroutines incrementen un comptador compartit.

package main

import (
    "fmt"
    "sync"
)

var (
    counter int
    mu      sync.Mutex
)

func increment(wg *sync.WaitGroup) {
    defer wg.Done()
    mu.Lock()
    counter++
    mu.Unlock()
}

func main() {
    var wg sync.WaitGroup

    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go increment(&wg)
    }

    wg.Wait()
    fmt.Println("Comptador Final:", counter)
}

Explicació

  • sync.WaitGroup: Utilitzat per esperar que totes les goroutines acabin.
  • mu.Lock() i mu.Unlock(): Asseguren que només una goroutine pugui incrementar el comptador a la vegada.
  • wg.Add(1) i wg.Done(): Gestionen el nombre de goroutines per esperar.

Millors Pràctiques per Utilitzar Mutexes

  • Minimitzar les Seccions Crítiques: Mantingues el codi dins de mu.Lock() i mu.Unlock() tan curt com sigui possible per reduir la contenció.
  • Evitar Deadlocks: Assegura't que tots els camins de codi desbloquegin el mutex, fins i tot en cas d'errors o pànics.
  • Utilitzar defer per Desbloquejar: Això assegura que el mutex es desbloquegi sempre, fins i tot si la funció retorna abans d'hora. 
    mu.Lock()
defer mu.Unlock()

Tècniques Avançades de Sincronització

Tot i que els mutexes són potents, Go proporciona altres primitius de sincronització que poden ser més adequats per a certs escenaris.

sync.RWMutex

Un RWMutex permet múltiples lectors o un sol escriptor, proporcionant millor rendiment per a càrregues de treball amb moltes lectures.

var rwMu sync.RWMutex

// Llegint
rwMu.RLock()
// Llegir recurs compartit
rwMu.RUnlock()

// Escriure
rwMu.Lock()
// Escriure recurs compartit
rwMu.Unlock()

sync.Cond

Un Cond (variable de condició) permet que les goroutines esperin o senyalin esdeveniments.

var cond = sync.NewCond(&sync.Mutex{})

func waitForCondition() {
    cond.L.Lock()
    cond.Wait()
    // Condició complerta, continuar
    cond.L.Unlock()
}

func signalCondition() {
    cond.L.Lock()
    cond.Signal()
    cond.L.Unlock()
}

Conclusió

Els mutexes són eines essencials per gestionar l'accés concurrent als recursos compartits en Go. Entenent com utilitzar sync.Mutex i sync.RWMutex, pots escriure programes concurrents segurs i eficients. Recorda seguir les millors pràctiques per evitar problemes comuns com els deadlocks i les condicions de carrera. A mesura que avancis, explora altres primitius de sincronització com sync.Cond per gestionar escenaris de sincronització més complexos.
