En aquesta secció, explorarem les diferències entre GraphQL i REST, dos enfocaments populars per construir APIs. Entendre aquestes diferències t'ajudarà a apreciar els avantatges únics de GraphQL i per què podria ser una millor opció per a certs casos d'ús.

Conceptes Clau

REST (Representational State Transfer)

  • Basat en recursos: Les APIs REST exposen recursos (per exemple, usuaris, publicacions) a través d'URLs.
  • Mètodes HTTP: Utilitza mètodes HTTP estàndard com GET, POST, PUT, DELETE.
  • Sense estat: Cada sol·licitud d'un client al servidor ha de contenir tota la informació que el servidor necessita per complir la sol·licitud.
  • Múltiples punts finals: Diferents punts finals per a diferents recursos i accions.

GraphQL

  • Llenguatge de consultes: Permet als clients sol·licitar exactament les dades que necessiten.
  • Punt final únic: Totes les sol·licituds s'envien a un únic punt final.
  • Fortament tipat: L'esquema defineix els tipus i les relacions en l'API.
  • Consultes flexibles: Els clients poden especificar la forma i la mida de la resposta.

Comparació

Estructura i Flexibilitat

| Característica | REST | GraphQL | |------------------|-------------------------------|-------------------------------| | Punt final | Múltiples | Únic | | Obtenció de dades| Estructura fixa per punt final| Flexible, definida pel client | | Sobre-obtenció | Comú | Evitat | | Sub-obtenció | Comú | Evitat | | Versionat | Sovint necessari | Normalment no necessari |

Exemple: Obtenció de dades d'usuari

REST

Per obtenir dades d'usuari i les seves publicacions, podries necessitar fer múltiples sol·licituds:

  1. Obtenir dades d'usuari:

    GET /users/1

    Resposta:

    {
  "id": 1,
  "name": "John Doe",
  "email": "john@example.com"
}

  2. Obtenir les publicacions de l'usuari:

    GET /users/1/posts

    Resposta:

    [
  {
    "id": 101,
    "title": "Post 1",
    "content": "Content of post 1"
  },
  {
    "id": 102,
    "title": "Post 2",
    "content": "Content of post 2"
  }
]

GraphQL

Amb GraphQL, pots obtenir les mateixes dades en una única sol·licitud:

query {
  user(id: 1) {
    id
    name
    email
    posts {
      id
      title
      content
    }
  }
}

Resposta:

{
  "data": {
    "user": {
      "id": 1,
      "name": "John Doe",
      "email": "john@example.com",
      "posts": [
        {
          "id": 101,
          "title": "Post 1",
          "content": "Content of post 1"
        },
        {
          "id": 102,
          "title": "Post 2",
          "content": "Content of post 2"
        }
      ]
    }
  }
}

Consideracions de Rendiment

  • REST: Múltiples punts finals poden portar a sobre-obtenció o sub-obtenció de dades, resultant en un ús ineficient de la xarxa.
  • GraphQL: Permet als clients sol·licitar només les dades que necessiten, reduint la quantitat de dades transferides per la xarxa.

Gestió d'Errors

  • REST: Utilitza codis d'estat HTTP estàndard (per exemple, 404 per no trobat, 500 per error del servidor).
  • GraphQL: Els errors formen part de la càrrega útil de la resposta, proporcionant informació detallada sobre què ha fallat.

Corba d'Aprenentatge

  • REST: Més fàcil de comprendre per a principiants a causa de la seva simplicitat i ús generalitzat.
  • GraphQL: Requereix entendre el seu esquema i llenguatge de consultes, que pot ser més complex inicialment.

Conclusió

Tant REST com GraphQL tenen els seus punts forts i febles. REST és simple i àmpliament adoptat, fent-lo una bona opció per a APIs senzilles. No obstant això, GraphQL ofereix més flexibilitat i eficiència, especialment per a aplicacions complexes on els clients necessiten obtenir diversos tipus de dades en una única sol·licitud.

Entenent les diferències entre aquests dos enfocaments, pots prendre una decisió informada sobre quin utilitzar per al teu proper projecte. A mesura que avancis en aquest curs, obtindràs una comprensió més profunda de com aprofitar GraphQL per construir APIs potents i eficients.
