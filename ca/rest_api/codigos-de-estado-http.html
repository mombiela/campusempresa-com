Introducció

Els codis d'estat HTTP són respostes estàndard proporcionades pels servidors web per indicar el resultat d'una sol·licitud HTTP. Aquests codis són essencials per al desenvolupament i la depuració d'APIs RESTful, ja que informen al client sobre l'èxit o el fracàs d'una sol·licitud.

Categories de Codis d'Estat HTTP

Els codis d'estat HTTP es divideixen en cinc categories principals:

  1. 1xx (Informatius): Indiquen que la sol·licitud ha estat rebuda i el procés continua.
  2. 2xx (Èxit): Indiquen que la sol·licitud va ser rebuda, entesa i acceptada amb èxit.
  3. 3xx (Redirecció): Indiquen que s'han de prendre accions addicionals per completar la sol·licitud.
  4. 4xx (Errors del Client): Indiquen que hi va haver un error en la sol·licitud del client.
  5. 5xx (Errors del Servidor): Indiquen que el servidor va fallar en complir amb una sol·licitud aparentment vàlida.

Codis d'Estat HTTP Comuns

2xx (Èxit)

  • 200 OK: La sol·licitud ha tingut èxit. La resposta depèn del mètode HTTP utilitzat. 
    HTTP/1.1 200 OK
Content-Type: application/json

{
  "message": "Sol·licitud exitosa"
}
  • 201 Created: La sol·licitud ha tingut èxit i s'ha creat un nou recurs. 
    HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": 123,
  "message": "Recurs creat exitosament"
}
  • 204 No Content: La sol·licitud ha tingut èxit, però no hi ha contingut que retornar. 
    HTTP/1.1 204 No Content

3xx (Redirecció)

  • 301 Moved Permanently: El recurs sol·licitat ha estat mogut permanentment a una nova URL. 
    HTTP/1.1 301 Moved Permanently
Location: https://nova-url.com/recurs
  • 302 Found: El recurs sol·licitat es troba temporalment en una URL diferent. 
    HTTP/1.1 302 Found
Location: https://temporal-url.com/recurs

4xx (Errors del Client)

  • 400 Bad Request: La sol·licitud no va poder ser entesa o va ser malformada. 
    HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error": "Sol·licitud malformada"
}
  • 401 Unauthorized: La sol·licitud requereix autenticació de l'usuari. 
    HTTP/1.1 401 Unauthorized
Content-Type: application/json

{
  "error": "Autenticació requerida"
}
  • 403 Forbidden: El servidor entén la sol·licitud, però es nega a autoritzar-la. 
    HTTP/1.1 403 Forbidden
Content-Type: application/json

{
  "error": "Accés denegat"
}
  • 404 Not Found: El recurs sol·licitat no es va poder trobar. 
    HTTP/1.1 404 Not Found
Content-Type: application/json

{
  "error": "Recurs no trobat"
}

5xx (Errors del Servidor)

  • 500 Internal Server Error: El servidor va trobar una condició inesperada que li va impedir complir amb la sol·licitud. 
    HTTP/1.1 500 Internal Server Error
Content-Type: application/json

{
  "error": "Error intern del servidor"
}
  • 502 Bad Gateway: El servidor, actuant com una porta d'enllaç o proxy, va rebre una resposta invàlida del servidor upstream. 
    HTTP/1.1 502 Bad Gateway
Content-Type: application/json

{
  "error": "Porta d'enllaç incorrecta"
}
  • 503 Service Unavailable: El servidor no està disponible temporalment, generalment a causa de manteniment o sobrecàrrega. 
    HTTP/1.1 503 Service Unavailable
Content-Type: application/json

{
  "error": "Servei no disponible"
}

Exemples Pràctics

Exemple d'una Sol·licitud GET Exitosa

GET /api/usuaris/1 HTTP/1.1
Host: exemple.com

Resposta:

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "nom": "Joan Pérez",
  "email": "joan.perez@exemple.com"
}

Exemple d'una Sol·licitud POST amb Creació de Recurs

POST /api/usuaris HTTP/1.1
Host: exemple.com
Content-Type: application/json

{
  "nom": "Anna Gómez",
  "email": "anna.gomez@exemple.com"
}

Resposta:

HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": 2,
  "nom": "Anna Gómez",
  "email": "anna.gomez@exemple.com"
}

Conclusió

Comprendre els codis d'estat HTTP és fonamental per al desenvolupament i la depuració d'APIs RESTful. Aquests codis proporcionen informació crucial sobre el resultat de les sol·licituds HTTP i ajuden a identificar i resoldre problemes de manera eficient. En familiaritzar-se amb els codis d'estat més comuns, els desenvolupadors poden millorar la comunicació entre el client i el servidor, assegurant una experiència d'usuari més fluida i efectiva.
