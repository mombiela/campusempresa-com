Introducció a REST

REST (Representational State Transfer) és un estil d'arquitectura per dissenyar serveis web. Va ser introduït per Roy Fielding en la seva tesi doctoral l'any 2000. REST es basa en un conjunt de principis que permeten la creació de serveis web escalables i fàcils de mantenir.

Principis Fonamentals de REST

1. Client-Servidor

  • Separació de Responsabilitats: L'arquitectura REST separa les responsabilitats del client i del servidor. El client és responsable de la interfície d'usuari i l'experiència de l'usuari, mentre que el servidor s'encarrega de la gestió de dades i la lògica de negoci.
  • Independència: Aquesta separació permet que ambdós components evolucionin de manera independent.

2. Stateless (Sense Estat)

  • Sense Estat: Cada sol·licitud del client al servidor ha de contenir tota la informació necessària per entendre i processar la sol·licitud. El servidor no emmagatzema cap estat del client entre les sol·licituds.
  • Escalabilitat: En no mantenir estat, els servidors poden gestionar més sol·licituds simultàniament, millorant l'escalabilitat.

3. Cacheable

  • Caché: Les respostes han de ser explícitament marcades com a cacheables o no cacheables. Això permet als clients emmagatzemar respostes i reutilitzar-les per a sol·licituds futures, reduint la càrrega en el servidor.
  • Eficiència: L'ús de caché millora l'eficiència i el rendiment del sistema.

4. Interfície Uniforme

  • Uniformitat: Una interfície uniforme simplifica i desacobla l'arquitectura, permetent que cada part del sistema evolucioni de manera independent.
  • Elements de la Interfície Uniforme:
    • Identificació de Recursos: Els recursos s'identifiquen mitjançant URLs.
    • Manipulació de Recursos a través de Representacions: Els clients interactuen amb els recursos utilitzant les seves representacions (per exemple, JSON, XML).
    • Missatges Autodescriptius: Cada missatge conté suficient informació per descriure com processar-lo.
    • Hipermèdia com el Motor de l'Estat de l'Aplicació (HATEOAS): Els clients descobreixen les accions disponibles a través d'hipervincles proporcionats en les respostes.

5. Sistema en Capes

  • Capes: L'arquitectura pot estar composta per múltiples capes, cadascuna amb una responsabilitat específica. Les capes poden incloure servidors de seguretat, servidors de balanceig de càrrega, servidors de caché, etc.
  • Independència: Les capes permeten que els components es desenvolupin i mantinguin de manera independent.

6. Codi Sota Demanda (Opcional)

  • Codi Sota Demanda: Els servidors poden estendre la funcionalitat del client enviant codi executable (per exemple, scripts JavaScript). Aquest principi és opcional i no és necessari per a tots els sistemes RESTful.

Exemples de Codi

Exemple d'una Sol·licitud GET

GET /api/v1/users/123 HTTP/1.1
Host: example.com
Accept: application/json

Exemple d'una Resposta GET

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "John Doe",
  "email": "john.doe@example.com"
}

Exemple d'una Sol·licitud POST

POST /api/v1/users HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "name": "Jane Doe",
  "email": "jane.doe@example.com"
}

Exemple d'una Resposta POST

HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": 124,
  "name": "Jane Doe",
  "email": "jane.doe@example.com"
}

Conclusió

Els principis de REST proporcionen una guia clara per dissenyar serveis web escalables, eficients i fàcils de mantenir. En seguir aquests principis, els desenvolupadors poden crear APIs que siguin intuïtives i fàcils d'utilitzar per als clients. La separació de responsabilitats, l'absència d'estat, l'ús de caché, una interfície uniforme, un sistema en capes i, opcionalment, el codi sota demanda, són els pilars que sustenten l'arquitectura RESTful.
